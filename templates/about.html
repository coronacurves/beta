History

This software was designed to visualize geographically-specific trends of the covid-19 epidemic, based on data compiled by Johns Hopkins University and the New York Times. It was developed by a member of the US military for personal convenience. The software is offered "as is" with absolutely no claim by any party as to its accuracy, suitability, safety, correctness, or any other characteristic. Use it entirely at your own risk. If you are ill or are concerned about someone who seems to be ill, consult a physician.

The software may be operated from the command line or via a web browser.

Instructions for web use

  1. On the home page choose a parameter to visualize. Both Johns Hopkins (JHU) and the New York Times (NYT) provide information on the number of covid-19 deaths and the number of identified new cases.
  2. Choose one or more geographical localities by checking the corresponding check box. The names of some geographical localities are links -- click on the link to show sub-localities that may be chosen.
  3. Click the "{{ BUTTON_TEXT }}" button to display the plots for each geographic locality. There is help on how to interpret the plots.

Architecture of web implementation

If implemented as a web application, the system will pull down updated copies of the epidemiological data whenever JHU or the NYT update the data on their web sites. (Webmasters must set up a cron job to enable the auto-updating.)

The web application will automatically use the latest downloaded copy of the epidemiological data.

Shortcomings of web operations, and room for improvement

  1. User interface
    2. Bare bones, but functional. Ripe with improvement possibilities.
  2. Scalability
    3. Scalability is limited for two main reasons.

    Flask is the default web server. Greater scalability and stability could be achieved using Gunicorn and Nginx.

    Matplotlib draws the plots. Matplotlib is not thread-safe, and so, to operate in a web environment, the software kicks off a new process in response to each plot request and blocks until the process completes. This architecture could obviously be improved.

  3. Output format
    4. Is not customizable. On the other hand, it does provide a single, fixed .png file that can be easily shared.

Shortcomings of data

Additional data sources could be added.

Assuming the new data source is a file to be downloaded from the Internet, you'll subclass the WebSource class in the file web_grab.py. You'll have to look at the subclasses related to JHU and the NYT to figure out the slots to specify and methods to define.
